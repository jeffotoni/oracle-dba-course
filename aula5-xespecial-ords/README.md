# Aula 5 Especial - Oracle ORDS

> Aula prática para subir `ORDS` com `Podman`, habilitar schema e objetos REST no Oracle e testar APIs HTTP/JSON diretamente sobre tabelas Oracle.

## Objetivo da aula

Nesta aula, o foco é mostrar o Oracle como plataforma de dados **e também como camada de API**.

O objetivo é:

- subir o `ORDS` com `Podman`;
- conectar o `ORDS` ao Oracle já rodando;
- criar um pequeno sistema didático com 3 tabelas;
- popular esse sistema com dados iniciais;
- habilitar schema e objetos REST;
- testar endpoints com `curl` e clientes de API;
- comparar `ORDS` com a API construída em Go na aula anterior.

## Ideia do sistema

Para esta aula, a proposta é usar um mini sistema de **catálogo de modelos + clientes + histórico de chamadas**.

Isso permite expor APIs REST simples e também mostrar relacionamento entre tabelas.

### Tabelas usadas

- `LLM_MODELS`
- `API_CLIENTS`
- `API_REQUESTS`

### Leitura do sistema

- `LLM_MODELS` = catálogo dos modelos de IA;
- `API_CLIENTS` = aplicações ou times que consomem a API;
- `API_REQUESTS` = histórico das requisições feitas por cliente para um modelo.

## Linha de raciocínio da aula

## Quick start

## 1. Subir Oracle

### Recomendação para a aula

```txt

```

## 2. Login no registry da Oracle

```bash
podman login container-registry.oracle.com
```

## 3. Baixar a imagem do ORDS

```bash
podman pull container-registry.oracle.com/database/ords:latest
```

## 4. Criar volume de configuração

```bash
podman volume create ords-config
```

## 5. Subir o ORDS

```bash
podman run -d \
  --name ords \
  -p 8181:8080 \
  -e DBHOST=host.containers.internal \
  -e DBPORT=1522 \
  -e DBSERVICENAME=FREEPDB1 \
  -e ORACLE_PWD=OraclePwd123 \
  -v ords-config:/etc/ords/config:Z \
  container-registry.oracle.com/database/ords:latest
```

## 6. Ver logs

```bash
podman logs -f ords
```

## 7. Entrar no container

```bash
podman exec -it ords bash
```

## 8. Instalação inicial do ORDS

Dentro do container:

```bash
ords --config /etc/ords/config install
```

## 12. Criar as tabelas da aula

Conectar como `JEFF` e executar:

```sql
CREATE TABLE llm_models (
  id                 NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  nome               VARCHAR2(100) NOT NULL,
  provider           VARCHAR2(100) NOT NULL,
  input_token_limit  NUMBER NOT NULL,
  output_token_limit NUMBER NOT NULL,
  ativo              CHAR(1) DEFAULT 'S' NOT NULL,
  created_at         TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL
);

CREATE TABLE api_clients (
  id                 NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  nome               VARCHAR2(100) NOT NULL,
  app_name           VARCHAR2(100) NOT NULL,
  email_responsavel  VARCHAR2(150) NOT NULL,
  api_key            VARCHAR2(100) NOT NULL,
  ativo              CHAR(1) DEFAULT 'S' NOT NULL,
  created_at         TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL
);

CREATE TABLE api_requests (
  id                 NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  client_id          NUMBER NOT NULL,
  model_id           NUMBER NOT NULL,
  prompt_chars       NUMBER NOT NULL,
  response_chars     NUMBER NOT NULL,
  status_code        NUMBER NOT NULL,
  latency_ms         NUMBER NOT NULL,
  created_at         TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
  CONSTRAINT fk_api_requests_client
    FOREIGN KEY (client_id) REFERENCES api_clients(id),
  CONSTRAINT fk_api_requests_model
    FOREIGN KEY (model_id) REFERENCES llm_models(id)
);
```

## 15. Habilitar o schema REST

Conectar como `JEFF` e executar:

```sql
BEGIN
  ORDS.ENABLE_SCHEMA(
    p_enabled             => TRUE,
    p_schema              => 'JEFF',
    p_url_mapping_type    => 'BASE_PATH',
    p_url_mapping_pattern => 'jeff',
    p_auto_rest_auth      => FALSE
  );
END;
/

COMMIT;
```

## 16. Habilitar as tabelas REST

## 19. O que observar na aula

- o Oracle continua sendo o banco;
- o `ORDS` passa a ser a camada HTTP/JSON;
- as tabelas viram recursos REST;
- não foi necessário escrever backend em Go para expor esses objetos;
- a modelagem da tabela impacta diretamente a API exposta.

## 20. Comparação didática com a aula anterior

### API em Go

- a aplicação define a rota;
- a aplicação define o handler;
- a aplicação controla o SQL;
- a aplicação decide a resposta HTTP.

### API com ORDS

- o Oracle já possui os dados;
- o `ORDS` expõe o schema e os objetos;
- o banco participa diretamente da API;
- o foco sai do backend custom e vai para o ecossistema Oracle.

## 21. Regra final da aula

```txt
Go + Oracle = a aplicação controla a API
ORDS + Oracle = o ecossistema Oracle expõe a API
```
